# 通用全栈 AI 工作流设计方案

> 创建日期: 2026-04-07
> 状态: 已设计，待落地
> 适用范围: Web / 桌面 / 后端 / 脚本仓库，不绑定单一项目

## 1. 目标

这套方案解决的不是“让 AI 多装几个工具”，而是建立一套稳定、可复用、可审计的全栈 AI 工作系统。

核心目标：

1. **提高准确性**：让 AI 对代码、依赖、运行态和远程仓库分别取真，降低幻觉。
2. **提高效率**：让常见任务都有固定入口，不每次从零拼工具。
3. **降低返工**：每个任务都必须形成验证结果和文档沉淀。
4. **便于扩展**：新增项目时只替换少量配置，不重做流程。

---

## 2. 设计原则

### 2.1 原生能力优先

如果宿主本身已经有稳定的 shell、文件读写、代码编辑和 web 搜索能力，就不要再额外堆同类 MCP。  
MCP 应优先补足宿主缺失的能力，而不是复制已有能力。

### 2.2 真相分层

不同问题应从不同数据源取真：

| 真相类型 | 首选来源 | 作用 |
|---|---|---|
| 本地代码事实 | Serena / `rg` / `git` | 谁定义、谁调用、影响面 |
| 第三方库事实 | 官方文档 / Context7 | API、版本差异、迁移说明 |
| 运行时事实 | Playwright / Chrome DevTools / 测试日志 | 真实行为、UI 问题、网络问题 |
| 远程协作事实 | GitHub MCP / `gh` | PR、Issue、CI、Actions |
| 项目约束事实 | `AGENTS.md` / `docs/roadmap.md` / 设计文档 | 规则、边界、阶段目标 |

### 2.3 任务必须闭环

每个任务必须走完完整闭环：

`需求进入 -> 建上下文 -> 实施 -> 验证 -> 文档同步`

没有验证和文档同步，就不算完成。

### 2.4 读写权限最小化

优先用只读检索工具收集信息，确认方向后再进入写路径。  
需要联网、浏览器自动化、远程仓库写操作时，尽量按场景分离凭据和权限。

### 2.5 一类问题一组默认工具

不要“所有工具一起开”。  
前端问题、后端问题、集成问题、评审问题，各自要有默认工具组合。

## 3. 推荐工具栈分层

## 3.1 L0：宿主原生层

这一层是所有工作流的底座，优先保证稳定。

| 工具 | 角色 | 结论 |
|---|---|---|
| Codex / Claude 这类代码代理 | 主执行器 | 必须 |
| shell / `rg` / `git` | 本地检索与执行 | 必须 |
| 内建 web 搜索 | 最新资料核验 | 必须 |
| 计划与步骤跟踪工具 | 控制复杂任务 | 必须 |

说明：

- 如果宿主已具备可靠 shell 和文件编辑能力，**不建议默认再加 filesystem MCP**。
- 如果宿主已具备 web 搜索，额外搜索 MCP 不是第一优先级。

## 3.2 L1：代码语义层

这一层负责把“字符串搜索”升级成“符号级理解”。

| 工具 | 角色 | 建议 |
|---|---|---|
| Serena | 符号检索、引用分析、项目记忆 | 强烈建议 |
| ast-grep | 结构化搜索和安全重构 | 强烈建议 |
| `rg` + `git diff` + `git log` | 快速补充 | 保留 |

适用：

- 重构
- 调用链分析
- 大模块理解
- 影响面评估

## 3.3 L2：外部文档层

这一层负责解决“本地代码里没有答案”的问题。

| 工具 | 角色 | 建议 |
|---|---|---|
| Context7 | 拉取最新库文档和示例 | 强烈建议 |
| 官方网站 / 官方仓库 | 最终取真源 | 必须 |
| skills 生态 | 按专项任务扩展能力 | 建议 |

适用：

- 新接 SDK / 框架
- 升级依赖
- API 兼容性核验
- 设计新集成方案

## 3.4 L3：运行态与浏览器层

这一层负责把“我觉得会这样”变成“页面和接口确实这样”。

| 工具 | 角色 | 建议 |
|---|---|---|
| Chrome DevTools MCP | 即时检查 DOM、网络、控制台、性能 | 强烈建议 |
| Playwright MCP | 可复现的页面操作和验收流 | 强烈建议 |
| 单元 / 集成 / E2E 测试 | 自动化验证 | 必须 |

分工：

- **DevTools** 适合诊断和即时观察。
- **Playwright** 适合复现、回归和脚本化验收。

两者不要互相替代，应该配合使用。

## 3.5 L4：远程协作层

这一层负责连接代码仓库、任务系统和交付系统。

| 工具 | 角色 | 建议 |
|---|---|---|
| GitHub MCP Server | PR / Issue / Actions / 代码搜索 | 强烈建议 |
| `gh` CLI | 补充脚本化操作 | 建议 |
| CI 平台日志接口 | 验证构建和发布问题 | 建议 |

适用：

- Review
- PR 自动化
- CI 失败追踪
- 远程仓库状态核对

## 3.6 L5：知识沉淀层

这一层决定 AI 能不能持续变快，而不是每次重新理解。

建议统一沉淀：

- `AGENTS.md`：仓库规则
- `docs/roadmap.md`：阶段目标与进度
- `docs/plans/YYYY-MM-DD-*.md`：设计决策
- `MODULE.md` 或精简模块文档：模块边界与业务约束
- 验证脚本与测试：最终事实载体

## 4. 默认工具组合

## 4.1 新功能开发

默认组合：

1. Serena / `rg` / `git`：摸清影响面
2. roadmap / 设计文档：确认目标与边界
3. Context7 + 官方文档：补第三方依赖事实
4. shell + 编辑：实现
5. 测试 + Playwright / DevTools：验证
6. 文档同步：更新 roadmap / plans

退出条件：

- 功能可运行
- 验证通过
- 文档同步完成

## 4.2 Bug 排查

默认组合：

1. Playwright / DevTools：先复现
2. Serena + `git log` / `git blame`：定位代码路径
3. shell / 测试：最小修复
4. 回归测试：避免复发
5. 文档记录：记下触发条件和修复边界

关键原则：

- **先复现，再修**
- **先加回归，再宣布修复**

## 4.3 重构

默认组合：

1. Serena：符号级影响面扫描
2. ast-grep：结构化查找和批量改写
3. 编译 / 测试：持续校验
4. 文档同步：更新模块边界和约束

关键原则：

- 重构必须优先保证行为不变
- 使用语义工具而不是全仓库字符串替换

## 4.4 前端交互 / UI 问题

默认组合：

1. DevTools：看 DOM、样式、请求、控制台
2. Playwright：把复现过程脚本化
3. 本地组件与样式源码：精确修改
4. 截图 / 验证步骤：形成验收依据

关键原则：

- UI 问题不要只看源码猜
- 先看真实页面，再决定改动

## 4.5 外部集成 / API 接入

默认组合：

1. 官方文档
2. Context7
3. 最小可运行示例
4. 集成测试 / 合约测试
5. 设计文档沉淀

关键原则：

- 外部 API 问题以官方资料为准
- 重要集成必须留最小验证脚本

## 4.6 Code Review

默认组合：

1. `git diff` / GitHub MCP
2. Serena：看调用链与影响面
3. 测试 / 构建 / 页面复现：验证风险
4. 风险优先输出 findings

关键原则：

- 先找 bug、回归和缺失验证
- 摘要放后面，发现放前面

## 5. 标准闭环流程

建议所有任务统一使用下面六步，不因任务大小改变结构：

### Step 1. Intake

判断任务类型：

- 新功能
- Bug
- 重构
- 评审
- 调研 / 选型

同时识别：

- 是否需要最新外部信息
- 是否需要浏览器验证
- 是否涉及远程仓库
- 是否涉及高风险改动

### Step 2. Context

只收集完成当前决策需要的上下文，不一次性把全部资料塞进上下文。

优先顺序：

1. 仓库规则与 roadmap
2. 目标模块代码与调用链
3. 第三方文档
4. 运行态证据

### Step 3. Decision

明确：

- 具体改动边界
- 需要使用的工具组合
- 验证方式
- 文档同步点

### Step 4. Execute

实施时遵循：

- 小步提交思路
- 优先局部修改
- 先做关键路径
- 避免无证据的大范围猜测式重写

### Step 5. Verify

至少满足以下之一：

- 编译 / 类型检查通过
- 单元 / 集成测试通过
- Playwright 流程通过
- DevTools 观测确认问题消失

高风险任务建议满足两项以上。

### Step 6. Record

记录三类信息：

1. 做了什么
2. 怎么验证的
3. 后续还需验证什么

## 6. 推荐的基础安装清单

建议按优先级分三批补齐。

## 6.1 第一批：立即补齐

1. Serena
2. Context7
3. Chrome DevTools MCP
4. Playwright MCP
5. Sequential Thinking

这是最小高价值组合，已经覆盖：

- 代码理解
- 外部文档
- 浏览器诊断
- 自动化验收
- 复杂任务拆解

## 6.2 第二批：协作增强

1. GitHub MCP Server
2. `gh` CLI 工作流
3. skills 生态中的专项技能

适合：

- 多 PR 并行
- 远程 review
- CI / Issue / Actions 联动

## 6.3 第三批：按需增强

1. sqlite / 数据库类 MCP
2. draw.io / 架构图工具
3. 文档生成技能
4. 项目内统一 MCP 包装器（如 scriptmgr）

适合：

- 数据排查频繁
- 架构设计文档较多
- 多项目共用一套命令入口

## 7. 不建议默认引入的做法

1. **重复能力的 MCP**  
   宿主已经有强 shell 和编辑能力时，再堆 filesystem MCP，收益通常不高。

2. **同类浏览器工具堆叠但没有分工**  
   DevTools 与 Playwright 可以同时存在，但必须明确诊断与回归的边界。

3. **把密钥直接写死在仓库或用户配置中**  
   应迁移到环境变量或密钥管理层。

4. **让 AI 只靠历史记忆做外部集成**  
   版本敏感问题必须重新查官方资料。

5. **没有验证脚本就宣布“已修复”**  
   这会让 AI 反复回到同一个问题。

## 8. 通用配置建议

## 8.1 Codex 侧建议

建议保留原生 shell / web / 编辑能力，再补以下 MCP：

- `chrome-devtools`
- `context7`
- `sequential-thinking`
- `playwright`
- `serena`
- `github`

建议命名固定，不随项目变化，便于跨仓库复用。

## 8.2 Claude 侧建议

建议与 Codex 尽量对齐，避免两个宿主的工具栈割裂。  
至少保持下面四项一致：

1. Serena
2. Playwright / Chrome DevTools
3. Context7
4. GitHub MCP

## 8.3 文档结构建议

任何项目都建议至少保留：

```text
docs/
  roadmap.md
  plans/
    YYYY-MM-DD-*.md
```

复杂项目再逐步增加：

```text
docs/
  modules/
    <module>/MODULE.md
  runbooks/
  checklists/
```

## 9. 建议的组织级工作方式

如果团队内同时使用多个 AI 宿主，建议采用下面的角色分工：

### 9.1 主执行代理

负责：

- 代码修改
- 本地验证
- 文档同步

要求：

- 端到端完成任务
- 输出验证结果

### 9.2 次审计代理

负责：

- 第二视角 review
- 风险复核
- 外部资料补证

要求：

- 默认只读
- 不与主执行代理抢写同一文件

### 9.3 自动化验收层

负责：

- 类型检查
- 单元测试
- E2E
- 构建检查

要求：

- 必须可重复执行
- 不依赖人工口头说明

## 10. 适配当前环境的落地顺序

基于当前机器的可见能力，建议按下面顺序推进：

### Phase A：统一基线

1. 对齐 Codex 与 Claude 的 MCP 栈
2. 补齐 Playwright MCP
3. 补齐 GitHub MCP
4. 清理配置中的明文密钥

### Phase B：形成默认工作流

1. 固定“新功能 / Bug / 重构 / Review”四套默认工具组合
2. 固定文档沉淀位置：`roadmap.md` + `docs/plans`
3. 固定验证出口：类型检查 / 测试 / Playwright

### Phase C：项目内统一入口

1. 评估是否用 `scriptmgr` 作为统一 MCP / 自动化入口
2. 将常用检查、回归、截图、文档同步做成标准命令
3. 让 AI 优先调用统一入口，而不是每次拼接散命令

---

## 11. 结论

优秀的全栈 AI 工作流不是“工具越多越好”，而是：

1. 宿主原生能力做底座
2. Serena 负责代码语义
3. Context7 与官方文档负责外部事实
4. DevTools + Playwright 负责运行态证据
5. GitHub MCP 负责协作与交付链路
6. roadmap / plans / 模块文档负责长期记忆

只要这六层稳定，换项目时只需要替换项目规则和验证脚本，不需要重建方法论。
